# 库存管理系统开发文档

## 1. 文档说明

本文档面向参与本项目开发、联调、维护的人员，说明项目技术栈、目录结构、启动方式、接口约定、数据库设计和常见开发注意事项。

项目采用前后端分离架构：

- 前端：Flutter，负责登录、库存管理、领取/归还、记录查询、管理员面板等界面。
- 后端：Node.js + Express，提供 REST API、JWT 鉴权、Excel 导入导出和 MySQL 数据访问。
- 数据库：MySQL，保存用户、库存物料、操作记录和借用状态。

## 2. 技术栈

| 模块 | 技术 |
| --- | --- |
| 前端 | Flutter 3.x、Dart |
| 前端依赖 | http、shared_preferences、file_picker、intl、url_launcher、desktop_drop、cross_file、google_fonts |
| 后端 | Node.js 18+、Express |
| 后端依赖 | mysql2、jsonwebtoken、bcryptjs、cors、multer、xlsx、nodemon |
| 数据库 | MySQL 8.0+ |
| 鉴权 | JWT Bearer Token |
| 文件处理 | multer 上传 Excel，xlsx 解析和导出 Excel |

## 4. 本地开发环境

### 4.1 前置要求

- Node.js 18 或以上
- npm
- MySQL 8.0 或以上
- Flutter 3.x
- Dart SDK，随 Flutter 安装
- Chrome 或 Windows 桌面运行环境，用于 Flutter 调试

### 4.2 初始化数据库

在项目根目录执行：

```bash
mysql -u root -p < database/init.sql
```

数据库默认名称为 `inventory`。后端数据库连接配置位于：

```text
server/config/db.js
```

本地开发时按自己的 MySQL 账号修改 `host`、`user`、`password`、`database`。生产或多人协作环境建议改为环境变量，不要把真实数据库密码提交到代码仓库。

### 4.3 启动后端

```bash
cd server
npm install
npm run dev
```

也可以使用普通启动：

```bash
npm start
```

默认监听地址：

```text
http://localhost:3000
```

启动成功后访问根接口应返回 API 信息：

```bash
curl http://localhost:3000/
```

### 4.4 启动前端

```bash
cd client/my_app
flutter pub get
flutter run -d windows
```

Web 调试：

```bash
flutter run -d chrome
```

前端 API 地址配置位于：

```text
client/my_app/lib/config/api.dart
```

默认值：

```dart
static const String baseUrl = 'http://localhost:3000';
```

如果后端部署到其他地址，需要同步修改该配置。

## 7. 接口约定

### 7.1 通用响应格式

JSON 接口通常返回：

```json
{
  "success": true,
  "data": {},
  "message": "操作成功"
}
```

失败时：

```json
{
  "success": false,
  "message": "错误原因"
}
```

需要登录的接口使用请求头：

```http
Authorization: Bearer <token>
```

Excel 导出接口也支持通过 query 传 token：

```text
/excel/export?token=<token>
```

### 7.2 用户接口

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| POST | `/user/register` | 公开 | 注册普通用户，成功后返回 token 和用户信息 |
| POST | `/user/login` | 公开 | 登录，成功后返回 token 和用户信息 |
| GET | `/user/list` | 超级管理员 | 获取用户列表 |
| PATCH | `/user/:id/role` | 超级管理员 | 将普通用户和管理员之间进行角色调整，不能修改超级管理员 |

### 7.3 库存接口

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| GET | `/material` | 登录用户 | 获取库存列表，并返回当前用户已借数量 `borrowed_count` |
| POST | `/material/add` | 管理员 | 新增库存；同名、品牌、型号、规格已存在时累加库存 |
| PUT | `/material/:id` | 管理员 | 更新库存信息 |
| DELETE | `/material/:id` | 管理员 | 删除库存 |
| POST | `/material/take` | 登录用户 | 领取物料；管理员直接通过，普通用户进入待审批 |
| POST | `/material/return` | 登录用户 | 归还物料；管理员直接通过，普通用户进入待审批 |
| GET | `/material/borrowed` | 登录用户 | 查询当前用户仍未归还的物料 |

### 7.4 记录接口

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| GET | `/record` | 登录用户 | 管理员查看全部记录，普通用户只看自己的记录 |
| GET | `/record/pending` | 管理员 | 获取待审批记录 |
| POST | `/record/approve/:id` | 管理员 | 审批通过，按记录类型更新库存和借用数量 |
| POST | `/record/reject/:id` | 管理员 | 审批拒绝 |

### 7.5 Excel 接口

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| POST | `/excel/import` | 管理员 | 上传 `.xlsx` 文件并导入库存 |
| GET | `/excel/export` | 管理员 | 导出库存 Excel 文件 |

Excel 导入字段支持中文或英文表头。代码当前识别的主要字段包括：

| 含义 | 支持表头 |
| --- | --- |
| 物料名称 | `物料名称`、`物品`、`name`、`Name` |
| 品牌 | `品牌`、`brand`、`Brand` |
| 型号 | `型号`、`model`、`Model` |
| 规格 | `规格`、`spec`、`Spec`、`specification` |
| 数量 | `数量`、`库存数量`、`quantity`、`Quantity` |
| 备注 | `备注`、`remark`、`Remark` |

## 8. 数据库设计

### 8.1 users

保存系统用户。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INT | 主键，自增 |
| username | VARCHAR(50) | 用户名，唯一 |
| password | VARCHAR(255) | bcrypt 加密后的密码 |
| role | VARCHAR(20) | 角色：`super_admin`、`admin`、`user` |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 8.2 materials

保存库存物料。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INT | 主键，自增 |
| name | VARCHAR(50) | 物料名称 |
| brand | VARCHAR(50) | 品牌 |
| model | VARCHAR(50) | 型号 |
| spec | VARCHAR(50) | 规格 |
| quantity | INT | 当前库存数量 |
| remark | VARCHAR(255) | 备注 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

唯一约束：

```sql
UNIQUE KEY unique_item (name, brand, model, spec)
```

该约束用于判定导入或新增时是否为同一种物料。

### 8.3 records

保存入库、领取、归还、导出等操作记录。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INT | 主键，自增 |
| user_id | INT | 操作用户 ID |
| material_id | INT | 物料 ID；导出记录可为空 |
| count | INT | 数量 |
| type | VARCHAR(10) | 类型：`in`、`out`、`export` |
| status | VARCHAR(10) | 状态：`pending`、`approved`、`rejected` |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

说明：

- `out` 表示领取。
- `in` 在新增库存时表示入库，在归还流程中表示归还。
- 普通用户提交领取/归还时先写入 `pending` 记录。
- 管理员提交领取/归还时直接写入 `approved` 并立即更新库存。

### 8.4 user_borrowed

代码中的领取、归还和库存列表会依赖 `user_borrowed` 表保存“某用户当前未归还数量”。当前 `database/init.sql` 已包含该表，新环境执行初始化脚本后即可支持 `/material`、`/material/take`、`/material/return`、`/material/borrowed` 等接口。

表结构：

```sql
CREATE TABLE IF NOT EXISTS user_borrowed (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  material_id INT NOT NULL,
  count INT NOT NULL DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY unique_user_material (user_id, material_id),
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (material_id) REFERENCES materials(id)
);
```

## 9. 核心业务流程

### 9.1 登录流程

1. 前端调用 `POST /user/login`，传入用户名和密码。
2. 后端读取用户并用 bcrypt 校验密码。
3. 校验成功后签发 JWT，payload 包含 `id`、`username`、`role`。
4. 前端通过 `ApiService.setToken` 或登录响应保存 token。
5. 后续接口统一携带 `Authorization: Bearer <token>`。

### 9.2 新增库存流程

1. 管理员调用 `POST /material/add`。
2. 后端按 `name + brand + model + spec` 查询是否已存在。
3. 已存在则累加 `quantity` 并更新备注。
4. 不存在则插入新物料。
5. 写入一条 `records`，类型为 `in`，状态为 `approved`。

### 9.3 领取流程

管理员领取：

1. 调用 `POST /material/take`。
2. 后端校验库存是否足够。
3. 直接扣减 `materials.quantity`。
4. 累加 `user_borrowed.count`。
5. 写入 `records`，类型为 `out`，状态为 `approved`。

普通用户领取：

1. 调用 `POST /material/take`。
2. 后端写入 `records`，类型为 `out`，状态为 `pending`。
3. 管理员在待审批列表中审批。
4. 审批通过时扣减库存并累加借用数量。
5. 库存不足时记录会被拒绝。

### 9.4 归还流程

管理员归还：

1. 调用 `POST /material/return`。
2. 后端查询当前用户 `user_borrowed.count`。
3. 按可归还数量增加库存。
4. 扣减 `user_borrowed.count`。
5. 写入 `records`，类型为 `in`，状态为 `approved`。

普通用户归还：

1. 调用 `POST /material/return`。
2. 后端写入 `records`，类型为 `in`，状态为 `pending`。
3. 管理员审批通过后增加库存并扣减借用数量。

### 9.5 Excel 导入流程

1. 前端使用 multipart/form-data 上传 `.xlsx` 文件，字段名为 `file`。
2. 后端使用 multer 保存临时文件。
3. 使用 xlsx 读取第一个工作表。
4. 逐行解析物料名称、品牌、型号、规格、数量、备注。
5. 已存在物料则累加库存，不存在则新增。
6. 每条新增或更新均写入入库记录。
7. 导入结束后删除临时上传文件。

### 9.6 Excel 导出流程

1. 管理员调用 `GET /excel/export`，通过 header 或 query 提供 token。
2. 后端校验 token 和管理员权限。
3. 查询全部库存。
4. 使用 xlsx 生成临时 Excel 文件。
5. 写入一条 `export` 类型记录。
6. 通过 `res.download` 返回文件，并在下载后延迟删除临时文件。

## 10. 开发规范

### 10.1 后端

- 新增需要登录的接口时，先挂载 `authMiddleware`。
- 新增管理员接口时，使用 `adminMiddleware`；仅超级管理员使用 `superAdminMiddleware`。
- SQL 查询统一使用 `db.execute(sql, params)` 参数化写法，避免 SQL 注入。
- 涉及库存数量和记录状态的接口，要同时考虑 `materials`、`records`、`user_borrowed` 三张表的一致性。
- 对外返回错误时使用统一格式：`{ success: false, message }`。
- 新增字段时同时更新数据库脚本、前端模型和 Excel 导入导出逻辑。
- JWT 密钥、数据库密码等敏感配置后续建议迁移到环境变量。

### 10.2 前端

- API 调用优先集中到 `lib/services/api_service.dart`，页面不要直接散落 HTTP 请求。
- 后端返回模型字段变更时，同步更新 `lib/models` 下的数据类。
- 角色判断使用模型中的 `isAdmin`、`isSuperAdmin`，避免页面内重复硬编码。
- 新增页面时优先复用 `theme` 和 `widgets` 中的样式与组件。
- 异步请求要处理 loading、empty、error 三种状态，避免页面卡死或无反馈。

### 10.3 数据库

- 库存物料以 `name + brand + model + spec` 作为同物料判定标准。
- `quantity` 不应小于 0；扣减库存前必须校验库存是否足够。
- 记录状态只使用 `pending`、`approved`、`rejected`。
- 记录类型只使用 `in`、`out`、`export`。
- 生产环境需要定期备份数据库，尤其是 `materials` 和 `records`。

## 11. 常见问题

### 11.1 前端请求失败

检查：

- 后端是否已启动在 `http://localhost:3000`。
- `client/my_app/lib/config/api.dart` 中的 `baseUrl` 是否正确。
- 浏览器或桌面端是否能访问后端地址。
- 需要登录的接口是否携带了 JWT token。

### 11.2 登录成功后接口返回 401

检查：

- token 是否已写入 `ApiService._token`。
- 请求头是否为 `Authorization: Bearer <token>`。
- token 是否过期。后端当前有效期为 24 小时。
- 后端 `JWT_SECRET` 是否发生变化。

### 11.3 库存列表或归还接口报错

检查数据库中是否存在 `user_borrowed` 表。当前代码查询库存时会左连接该表，并在领取/归还时更新该表。

### 11.4 Excel 导入失败

检查：

- 文件格式是否为 `.xlsx`。
- 上传字段名是否为 `file`。
- 表头是否包含物料名称、品牌、规格和数量。
- 后端 `server/uploads` 是否有写入权限。

### 11.5 Excel 导出失败

检查：

- 当前用户是否为 `admin` 或 `super_admin`。
- URL query 或请求头中是否带 token。
- 后端 `server/exports` 是否有写入权限。

## 12. 后续优化建议

- 将数据库连接、JWT 密钥、端口号迁移到 `.env`。
- 为 `database/init.sql` 增加更完整的重复执行兼容性，降低二次初始化时的冲突风险。
- 将 `routes` 中的业务逻辑拆分到 controller/service 层。
- 为库存扣减、审批通过等关键流程增加数据库事务。
- 增加后端自动化测试，覆盖登录、权限、领取、归还、审批和 Excel 导入。
- 后续接口、表结构或部署流程变化时，同步更新 `docs/接口文档.md`、`docs/数据库设计.md`、`docs/部署说明.md`。
